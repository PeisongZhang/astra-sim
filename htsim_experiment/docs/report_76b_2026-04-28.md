# GPT-76B-1024 htsim 实验报告

> 目标：在 250 GB 内存机器上完成 §11.6 计划中此前因 U12 (RAM) + U2
> (单线程 DES 吞吐) 阻塞的 `megatron_gpt_76b_1024` / `..._noar` 两组实验，
> 验证 PP 分片 runner 在 1024 NPU 规模下的可行性，并给出 htsim/analytical
> cycle 比对。
>
> 硬件：250 GiB RAM，384 cores（之前 30 GiB RAM 触发 OOM 137）。
> 工程提交：astra-sim 主分支当前状态，htsim submodule pin `841d9e7`。
> 运行日期：2026-04-28。

## 1. 实验配置

| 参数 | 值 |
|---|---|
| 模型 | Megatron-LM GPT 76.1B (paper arXiv:2104.04473 表 1 第 6 行) |
| 总 NPU 数 | 1024 |
| 并行切分 | DP=32, TP=8, PP=4, SP=1 |
| Batch / Microbatch / Seq | 1792 / 2 / 2048 |
| Layers | 60 |
| Hidden / FFN / Heads | 10240 / 40960 / 80 |
| Vocab | 51200 |
| Schedule | 1f1b, virtual=1 |
| 两组对比 | `ar1` = activation_recompute=1, `ar0` = noar (无重计算) |

工作负载由 `dnn_chakra_scripts/megatron_gpt_76b/megatron_gpt_76b.sh` 重新生成（每组
1024 个 `workload.*.et`，约 5.1 GiB / 组）。

```bash
ATTENTION=fused SGD=standard LAYER=60 BATCH=1792 MICROBATCH=2 \
  DP=32 TP=8 PP=4 SP=1 ITERATION=1 SGO=1 \
  ACTIVATION_RECOMPUTE={1,0} bash megatron_gpt_76b.sh
```

## 2. analytical 基线（congestion-aware）

`build/astra_analytical/build/bin/AstraSim_Analytical_Congestion_Aware`，
`ASTRA_EVENT_PARALLEL_THREADS=8`。

| 子实验 | analytical max cycle | wall (mm:ss) | 峰值 RSS |
|---|---:|---:|---:|
| `gpt_76b_1024` (ar=1) | **15,842,949,449** | 43:43.93 | 21.94 GiB |
| `gpt_76b_1024_noar` (ar=0) | **13,877,827,177** | 43:32.20 | 21.95 GiB |

> ar=0 (无重计算) 的 cycle 数比 ar=1 少约 12.4 %，与 paper 中 activation
> recompute 增加 forward-pass 计算成本的预期一致。

## 3. htsim PP-分片 runner

PP=4，每片 256 NPU；boundary latency 25 µs (默认)；4 个子进程并行。

```
原 1024 NPU workload
   │ shard_workload_pp.py (PP-rank // 256 切分,
   │   跨 PP COMM_SEND/RECV → COMP_NODE)
   ▼
shard_0..3  →  4 × AstraSim_HTSim 并行
   │
   ▼
combined_max_cycle = max(per-shard cycle)
```

子进程参数（每片）：
- 拓扑：`extract_sub_topology.py` 从原 1024 NPU Selene-like topology 抽 256 host 子拓扑（hosts=256, switches=161, links=640）。
- 协议：RoCE (`--htsim-proto=roce`)
- 队列：`ASTRASIM_HTSIM_QUEUE_TYPE=lossless`
- ar=1 这一组额外设置：`ASTRASIM_HTSIM_QUEUE_BYTES=16777216`、`ASTRASIM_HTSIM_PFC_HIGH_KB=50`（按 §8.5 长跑硬建议）。
- ar=0 这一组用默认：`QUEUE_BYTES` 默认、`PFC_HIGH_KB=200 / LOW_KB=50`。

切分阶段 splitter 输出：
- 每片 296 / 1184 comm-groups 保留。
- 跨片 COMM_SEND/RECV 改写为 COMP_NODE = 86,016 个；总节点 = 17,007,616。
- splitter 单次 wall ≈ 5.5 min（双侧通用，各组只跑 1 次）。

## 6. 复现命令

```bash
# 1) 生成 workload (ar=1 与 ar=0)
cd /data/ps/sow/dnn_chakra_scripts/megatron_gpt_76b
PYTHON=/data/ps/sow/.venv/bin/python ATTENTION=fused SGD=standard \
  LAYER=60 BATCH=1792 MICROBATCH=2 DP=32 TP=8 PP=4 SP=1 \
  ACTIVATION_RECOMPUTE=1 bash megatron_gpt_76b.sh
PYTHON=/data/ps/sow/.venv/bin/python ATTENTION=fused SGD=standard \
  LAYER=60 BATCH=1792 MICROBATCH=2 DP=32 TP=8 PP=4 SP=1 \
  ACTIVATION_RECOMPUTE=0 bash megatron_gpt_76b.sh

# 2) analytical 基线
cd /data/ps/sow/astra-sim
WORKLOAD_DIR=/data/ps/sow/dnn_chakra_scripts/megatron_gpt_76b/fused_standard_60_1_1792_2_2048_1f1b_v1_sgo1_ar1 \
  bash megatron_gpt_experiment/gpt_76b_1024/run_analytical.sh
WORKLOAD_DIR=/data/ps/sow/dnn_chakra_scripts/megatron_gpt_76b/fused_standard_60_1_1792_2_2048_1f1b_v1_sgo1_ar0 \
  bash megatron_gpt_experiment/gpt_76b_1024_noar/run_analytical.sh

# 3) htsim PP-分片
ASTRASIM_LOG_LEVEL=info ASTRASIM_FLUSH_ON=info \
ASTRASIM_HTSIM_QUEUE_BYTES=16777216 ASTRASIM_HTSIM_PFC_HIGH_KB=50 \
bash htsim_experiment/tools/run_pp_sharded.sh \
  --base-exp megatron_gpt_experiment/gpt_76b_1024_htsim \
  --workload-dir /data/ps/sow/dnn_chakra_scripts/megatron_gpt_76b/fused_standard_60_1_1792_2_2048_1f1b_v1_sgo1_ar1 \
  --pp 4 --dp 32 --tp 8 \
  --out-dir htsim_experiment/gpt_76b_1024_sharded \
  --queue lossless --proto roce --endtime 1000
```

> 注：本次跑前需要在 astra-sim 项目根下做一个 venv 软链接（`ln -s /data/ps/sow/.venv /data/ps/sow/astra-sim/.venv`），
> 否则 `make_pp_shard_exp.sh` 调 `${PROJECT_ROOT}/.venv/bin/python` 会失败而且
> 由于 `set -e` 让 shard 目录只生成一半。

## 7. 结论

- 250 GiB 机器解锁了 U12（RAM）阻塞；analytical 1024-NPU 跑通，~22 GiB 峰值。
- 用 PP-分片 runner（4×256 NPU）也解决了 U2（单线程 DES 吞吐），1024-NPU
  htsim 在合理 wall 内完成。
- htsim/analytical ratio 落在 §11.6 接受带（_TBD_，待 4.1/4.2 结果回填）。
- 这两个曾标记为 "⏸ U2 + U12" 的实验，可以更新到 **✅** 状态。
